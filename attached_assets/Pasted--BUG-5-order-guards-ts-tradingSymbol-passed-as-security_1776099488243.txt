BUG #5 — order-guards.ts: tradingSymbol passed as securityId (blacklist never works)
FILE: artifacts/api-server/src/routes/orders.ts + artifacts/api-server/src/lib/order-guards.ts
PROBLEM: In orders.ts, runOrderGuards is called with:
  tradingSymbol: String(parsed.data.securityId)
This passes the numeric security ID (e.g., "1333") as the symbol. But users configure
the blacklist with trading symbols like "NIFTY", "BANKNIFTY". The blacklist check will
NEVER match because it compares "1333".includes("NIFTY") = false.
FIX: The order payload needs to include the trading symbol. Either:
  Option A: Add tradingSymbol as an optional field in PlaceOrderBody zod schema and
  pass it through from the frontend when available.
  Option B: In checkInstrumentBlacklist(), also check against the securityId if provided.
  Best fix: Update PlaceOrderBody to accept an optional tradingSymbol field, and
  pass parsed.data.tradingSymbol ?? String(parsed.data.securityId) to runOrderGuards.

BUG #6 — order-guards.ts: maxDailyLoss and maxDailyProfit limits are NEVER enforced
FILE: artifacts/api-server/src/lib/order-guards.ts
PROBLEM: Settings has maxDailyLoss and maxDailyProfit fields, and the UI shows them in
Risk Manager. But runOrderGuards() never checks these limits. Orders keep getting placed
even when the daily loss limit is breached.
FIX: Add a new guard function checkDailyPnlLimits() and call it in runOrderGuards():

  export async function checkDailyPnlLimits(): Promise<OrderGuardResult> {
    const settings = await getSettings();
    const maxLoss = settings?.maxDailyLoss ? Number(settings.maxDailyLoss) : null;
    const maxProfit = settings?.maxDailyProfit ? Number(settings.maxDailyProfit) : null;
    if (!maxLoss && !maxProfit) return { allowed: true };

    try {
      const positions = await dhanClient.getPositions() as Array<Record<string, unknown>>;
      const todayPnl = Array.isArray(positions)
        ? positions.reduce((s, p) => s + Number(p.realizedProfit || 0) + Number(p.unrealizedProfit || 0), 0)
        : 0;

      if (maxLoss && todayPnl <= -maxLoss) {
        return { allowed: false, reason: `Daily loss limit of ₹${maxLoss.toLocaleString("en-IN")} reached. Today's P&L: ₹${todayPnl.toFixed(2)}` };
      }
      if (maxProfit && todayPnl >= maxProfit) {
        return { allowed: false, reason: `Daily profit target of ₹${maxProfit.toLocaleString("en-IN")} reached. Today's P&L: ₹${todayPnl.toFixed(2)}` };
      }
    } catch {
      return { allowed: true };
    }
    return { allowed: true };
  }

Then call it inside runOrderGuards() before the position size check.

BUG #7 — super-orders.ts: Target price and stop loss are stored but NEVER monitored
FILE: artifacts/api-server/src/routes/super-orders.ts
PROBLEM: The super order system stores targetPrice and stopLossPrice in the database,
and the UI shows these fields. But there is ZERO background monitoring. No job watches
open positions and places exit orders when the target or stop loss is hit. These fields
are purely cosmetic — they do nothing.
FIX: Create a new file: artifacts/api-server/src/lib/super-order-monitor.ts

  This monitor should:
  1. Run on a setInterval (every 5 seconds during market hours)
  2. Fetch all super orders from DB where status is NOT 'CANCELLED' and NOT 'COMPLETED'
  3. For each super order that has a dhanOrderId, check the live LTP via dhanClient.getLtp()
  4. If LTP >= targetPrice → place a SELL/BUY (opposite) order to exit, update DB status to 'TARGET_HIT'
  5. If LTP <= stopLossPrice (for BUY) or LTP >= stopLossPrice (for SELL) → place exit order, update DB status to 'STOP_LOSS_HIT'
  6. Send Telegram alert for target hit or stop loss hit
  7. Only run between 09:15 and 15:25 IST on weekdays
  Import and call startSuperOrderMonitor() in index.ts alongside startAutoSquareOffScheduler().
