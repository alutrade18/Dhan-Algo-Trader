Full Project Audit — Issue List
🔴 CRITICAL — Crashing Right Now in Production
#1 — OrderUpdateWS infinite crash-loop (confirmed in logs, every ~5 sec)

File: api-server/src/lib/order-update-ws.ts line 63
Problem: Dhan sends a binary control frame ("2\n\u0000(\u0003") immediately after your auth JSON. JSON.parse throws a SyntaxError on it. The catch block logs the warning, but Dhan then closes the socket (code 1006) — probably because our server isn't acknowledging its handshake. Result: connect → auth → crash → disconnect → reconnect, endlessly, forever.
Impact: No live order status updates. Every single order placed has no real-time confirmation over WebSocket.
#2 — MarketFeedWS instant disconnect (code 1006 immediately after connect)

File: api-server/src/lib/market-feed-ws.ts
Problem: Connects to wss://api-feed.dhan.co?version=2&token=... and disconnects within 250ms every time. No error message — just code 1006 (abnormal closure). Either Dhan rejects the auth token via the URL, or the binary handshake packet (response code in first byte) isn't being acknowledged before they drop us.
Impact: Zero live price ticks ever received. The Option Chain, Positions LTP, and Super Order price feed are all dead.
🟠 HIGH — Wrong Behavior, No Crash
#3 — Kill Switch NOT enforced on server-side order placement

Files: routes/orders.ts, lib/order-guards.ts
Problem: runOrderGuards() only checks the daily loss limit. It does NOT check if the kill switch is active. A direct API call to POST /api/orders bypasses the kill switch entirely. The frontend blocks the UI, but the backend doesn't.
Impact: The kill switch safety net has a hole — any direct HTTP request places live orders regardless.
#4 — notificationPreferences field does not exist in DB schema

File: lib/auto-square-off.ts line ~55
Problem: settings.notificationPreferences as { autoSquareOff?: boolean } — that field is not in the settingsTable schema at all. The as cast silences the TypeScript error. The value is always undefined, so prefs?.autoSquareOff !== false is always true. The Telegram auto-square-off alert can never be disabled.
Impact: Telegram notifications cannot be turned off for auto-square-off events.
#5 — activeStrategies hardcoded to 0 in dashboard API

File: routes/dashboard.ts
Problem: activeStrategies: 0 is a literal constant. Dashboard "Active Strategies" card always shows 0 regardless of what's in the strategies table.
Impact: Dashboard metric is permanently wrong.
🟡 MEDIUM — Suboptimal, Edge-Case Wrong Behavior
#6 — Kill switch not auto-reset across accounts (broker disconnect)

File: routes/broker.ts
Problem: When you disconnect a broker and connect a new one, the kill switch state (both Dhan-side and DB) is not reset. The new account could inherit an active kill switch from the old session.
#7 — Super Orders: INTRADAY-only rule not enforced on backend

File: routes/super-orders.ts
Problem: Project spec says "Super Orders are INTRADAY only." The backend accepts any productType with no validation. A crafted API request can place a CNC super order.
#8 — /dashboard/recent-activity has zero caching

File: routes/dashboard.ts
Problem: Every frontend poll calls dhanClient.getOrders() fresh — no in-memory cache. Burns Dhan data API rate limit (1000/min). If the frontend polls every 10 seconds with multiple tabs open, this alone can exhaust the API budget.
#9 — Order guards call getPositions() on every single order placement

File: lib/order-guards.ts
Problem: checkDailyLossLimit does a live Dhan API call (getPositions) AND a DB query synchronously during order placement. If Dhan is slow or rate-limited, this delays every order. Should use a short-lived cache.
#10 — auto-square-off.ts uses console.log instead of the structured logger

File: lib/auto-square-off.ts
Problem: All log lines use console.log / console.error. These don't appear in the Pino JSON log, and are invisible on the Logs page in the frontend.
#11 — market-calendar.ts also uses console.info/console.warn

File: lib/market-calendar.ts
Problem: Same issue — holiday loading events are invisible in the structured logs.
#12 — MarketFeedWS subscribe() overwrites existing subscriptions for same segment

File: lib/market-feed-ws.ts
Problem: The subscription map key is ${exchangeSegment}:${mode}. Calling subscribe("NSE_EQ", [1333]) then subscribe("NSE_EQ", [500209]) replaces the first entry. On reconnect, [1333] never re-subscribes. Silent data loss for multi-position scenarios.
⚪ LOW — Cosmetic, Dead Code, Minor
#13 — dashboardWidgets.totalPnl and .equityCurve are dead config entries

File: lib/db/src/schema/settings.ts
Problem: These two widget flags exist in the schema and DB but are now meaningless — both features were moved to the Ledger Statement page. The settings UI still shows "Total P&L" widget toggle, which does nothing.
#14 — winRate hardcoded to 0 in dashboard API

File: routes/dashboard.ts
Problem: winRate: 0 always returned. The Active Strategies card shows "Win Rate: 0.0%" permanently.
#15 — Equity scheduler doesn't skip NSE market holidays

File: lib/equity-scheduler.ts
Problem: Only skips weekends via isWeekendIST(). On NSE holidays (Diwali, Republic Day, etc.) it still fires the 15:30 IST job and hits Dhan's ledger API unnecessarily.
#16 — Rate-limit daily counter DB writes are fire-and-forget (silent failure)

File: lib/rate-limiter.ts
Problem: void this.upsertToDb(...) — if the DB write fails, the in-memory counter is correct but not persisted. On server restart, the daily order counter resets to 0.
#17 — FRONTEND_URL env var referenced but never set

File: src/app.ts
Problem: process.env.FRONTEND_URL is in the CORS allowlist but the env var doesn't exist. It's undefined, filtered out by .filter(Boolean). Dead code.
#18 — Postback endpoint has no IP allowlist, secret is optional

File: routes/postback.ts
Problem: If POSTBACK_SECRET env var is not set, any anonymous caller can POST to /api/postback and trigger Telegram alerts and Socket.IO order events. In production this is an open webhook.
#19 — Backtesting page has an empty description <p> tag

File: pages/backtesting.tsx
Problem: The description paragraph is literally empty — looks incomplete.
#20 — PIN_SALT hardcoded fallback in settings

File: routes/settings.ts
Problem: process.env.PIN_SALT ?? "rajesh-algo-pin-salt" — if the env var is not set, all deployments share the same PIN hash salt. Minor security weakness.
